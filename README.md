## 📦 这是什么

一个 16 位无符号 ALU，支持加减乘除、逻辑、移位、比较，共 15 种操作。4 个子模块各自独立，顶层用 case 分支做操作码译码。

**能跑，能出波形，课程设计通过了。** 但回头看，到处都是新手坑。

---

## 🪵 踩过的坑

### 1. 复制粘贴的后果

所有文件的头注释里都写着：

```
// cinreated On    : 2024-06-05
// Descinription   :
```

`Created` 拼成了 `cinreated`，`Description` 拼成了 `Descinription`。显然第一份写错了之后，后面 9 个文件全是复制粘贴的——**一个都没改过来**。

更要命的是顶层 testbench：

```verilog
// cinreated On    : 4024-06-07
```

年份写成了 **4024** 年。也是复制粘贴没改。

> **教训**：复制粘贴省了 3 秒，但当你需要向别人展示代码的时候，每个拼写错误都在说 "我不在乎细节"。现在写代码，头注释要么不写，要么一次写对，不复制。

---

### 2. 同一个模块定义了两次

`full_adder` 的 Verilog 代码出现在两个文件里：
- `mini_ALU_16bit_ADD.v` 第 57-67 行
- `mini_ALU_16bit_SUB.v` 第 80-90 行

**逻辑完全一样**。`mini_ALU_16bit_ADD` 这个模块也被定义了两次——但在 `SUB.v` 里端口列表不一样（`overflow + valid` vs `cout`）。

如果你把两个文件都加进同一个 Vivado 工程，综合器会报 `module redefinition` 错误。当时能跑通，大概率是因为仿真时只编译了用到的文件。

> **教训**：公共模块抽出来，一个文件只定义一个模块。`full_adder` 应该放在 `full_adder.v` 里，然后一起编译。不是所有能跑的东西都是对的——可能只是你运气好。

---

### 3. 加法器的性能问题（当时完全没概念）

```verilog
// 行波进位：进位从 bit0 一路传到 bit15
full_adder fa0(...);   // bit 0 先算
full_adder fa1(...);   // bit 1 等 bit0 的进位
// ...
full_adder fa15(...);  // bit 15 等前面 15 级的进位
```

16 个全加器串在一起，最长路径是进位从 bit0 传到 bit15——**16 级门延迟**。位数翻倍，延迟翻倍。

当时不知道这叫「行波进位」，也不知道工业界用的是超前进位（Carry Lookahead Adder），更不知道 `+` 运算符让综合器自动优化比手写行波进位好得多。

> **教训**：`assign sum = a + b` 在 90% 的场景下比你手写的加法器更快、更省面积、更不出错。手写全加器链是课本教的东西，用来理解原理可以，用来做工程不行。**下次先查 "Verilog best practice for adder"，不要上来就写全加器。**

---

### 4. 乘法器的面积灾难

乘法器逻辑拆开看：

```
第 1 步：生成 16 个部分积（移位后的 data0）
第 2 步：用 15 个 31-bit 加法器串行累加
```

相当于把 16 个数从头加到尾，**纯组合逻辑**，没有流水线，没有压缩。综合出来的电路是一个巨大的组合逻辑云，面积大、频率低。

当时在注释里写了 "可优化为 Wallace Tree 或 Booth 编码"——但那是从课本上抄的，**实际上并不知道 Wallace Tree 怎么实现**。

> **教训**：注释里写优化方案不等于你理解了它。如果今天重写这个乘法器，要么用 `assign product = data0 * data1`（让工具做优化），要么真正手写一个 3-2 压缩器 + 超前进位加法器的 Wallace Tree。**不要写你自己解释不清楚的注释。**

---

### 5. 除法器——唯一有时序逻辑的模块，也是最难的

除法用了两段式状态机 + 恢复余数算法，16 个周期出结果。当时调了好久，因为：

- 状态机刚开始写了一段式（`always @(posedge clk)` 里既算 next_state 又更新状态），逻辑混在一起
- 后来改成两段式（组合逻辑算 next_state + 时序逻辑更新寄存器），才勉强调通
- `Z` 的位宽是 `[32:0]`（33 位），但实际只用 32 位——多了一位，纯属心虚多留的

最大的问题是：**写完了但说不清楚 "恢复余数" 每一步在做什么**。考试被问到的时候磕磕绊绊。

> **教训**：能跑 ≠ 真懂。如果一个模块你没法在纸上一步步画出数据流，你就是没懂。**下次先手算 3 遍，画时序图，再写代码。**

### 6. 减法器的 "溢出" 语义混乱

```verilog
assign diff = (data0 < data1) ? (~result_register + 1) : result_register;
assign overflow = (data0 < data1) ? 1'b1 : 1'b0;
assign valid = (data0 < data1) ? 1'b0 : 1'b1;
```

当 `data0 < data1` 时（比如 5 - 10），结果取绝对值（输出 5），同时 `overflow=1, valid=0`。

这里的问题是：**对无符号减法来说，"溢出" 的定义根本不应该是 "结果为负"**。无符号减法在硬件里用补码实现，5 - 10 在 16 位下会得到 65531（wraparound），这是正确的无符号行为，不是溢出。

实际上这里把减法做成了 "当被减数不够时返回绝对值并报错"——这是**计算器**的逻辑，不是 ALU 的逻辑。

> **教训**：定义接口之前先想清楚 "这个信号到底表示什么"。overflow 在数字电路里有严格定义（有符号数超出范围才叫溢出），不要随便复用已有的概念去表达别的意思。**如果你发现自己在用一个不合适的词凑合，那就造一个新词，或者重新设计。**

### 7. Testbench 全部只靠人眼看波形

所有 testbench 的结构都是：

```verilog
initial begin
    data0 = xxx; data1 = xxx;
    #40 data0 = yyy; data1 = yyy;
    // ...
end
initial $monitor(...);  // 打印到控制台
initial $dumpfile(...); // 导出波形文件
```

然后**对着波形截图手动判断对不对**。没有一行代码在自动校验结果。

正确做法应该是：

```verilog
// 自动检查
always @(posedge clk) begin
    if (valid) begin
        if (OP == ADD) assert(result == data0 + data1) else $error("ADD failed");
    end
end
```

> **教训**：手动看波形在 5 个测试向量时还行，50 个就不行了，500 个就不可能。**testbench 的本质是用代码验证代码**。下次哪怕只写一个 `if (expected != actual) $error(...)` 也比全靠眼睛强。学了 SystemVerilog 之后用 `assert` 会更方便。

### 8. 小的东西

- **`timescale 1ns/1ns`**：时间精度和单位一样，理论上没问题但实际应该给更高精度（`1ns/1ps`），因为门延迟通常在 ps 级别
- **`num_shift` 声明为 5-bit 但 testbench 里写 `4'd3`**（4 位宽？5 位宽？随意了）
- **SUB testbench 文件名和头注释对不上**：文件名是 `ADD_tb.v`，头注释写 `MUL_tb.v`
- **除法器 `@valid` 等待**：顶层 testbench 用了 `@valid` 来等除法完成，这个写法是对的——但别的模块都没用这种同步方式，风格不统一
- **时钟和复位**：除了除法器，其他模块都是纯组合逻辑——但顶层却用了 `always @(posedge clk)` 来锁存结果。这导致时钟和复位只在除法时有意义，其他操作时纯属多余的电平变化

## ✅ 做对了的事

说了一堆问题，也有做得好的：

| 做对了什么 | 为什么重要 |
|-----------|-----------|
| **每个子模块单独写** | 没把所有东西塞进一个文件，结构清晰，方便分别调试 |
| **top-down 设计** | 先定操作码，再定接口，再写子模块——思路是对的 |
| **每个模块都写了 testbench** | 虽然不够自动化，但测试意识已经有了 |
| **用 generate 生成加法器链** | 学会了参数化生成，比手写 16 个 instantiation 强 |
| **除法器用状态机** | 第一次写时序逻辑就用了标准的两段式，规范 |
| **存了波形截图** | 至少知道验证结果要留档 |
| **写注释标了优化方向** | 虽然当时没能力实现，但知道自己的代码不完美 |

## 🔁 如果重来，我会怎么做

1. **先读 3 个开源 ALU 项目**，搞懂别人怎么组织代码，而不是闷头就写
2. **用 `assign result = a + b` 而不是手写全加器**，把时间花在理解时序和验证上
3. **写自检查 testbench**，用随机向量 + 自动比对，而不是 `$monitor` + 截图
4. **乘法器和除法器各画一张数据流图**，确认每个周期每个寄存器在干什么
5. **统一代码风格**：一个文件一个模块，信号命名用 snake_case，注释用英文（不容易乱码）
6. **用 Git 记录调试过程**：改了什么、为什么改、改完对不对——而不是一个 commit 塞所有修改
